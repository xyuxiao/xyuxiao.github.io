OkHttp源码阅读-拦截器

发表于 2019-06-19 | 更新于 2019-07-08 | 分类于 Android | 阅读次数：

执行顺序

不管是同步还是异步任务最后都会走到getResponseWithInterceptorChain方法中,在这个方法里会创建一个List用于存放一系列的拦截器,包括OkHttp官方定义的,也包括自定义的拦截器,这方法会递归执行所有拦截器在最后CallServerInterceptor中进行数据请求

从这代码可以看出来执行顺序是interceptors->RetryAndFollowUpInterceptor->BridgeInterceptor->CacheInterceptor->ConnectInterceptor->networkInterceptors->CallServerInterceptor

拦截器

Request

RetryAndFollowUpInterceptor

先从chain里获取出来request,然后会调用transmitter.prepareToConnect(request)

这边会判断是否已经存在一个连接,如果已经存在并且存在连接和本次想要建立的连接是同一个则直接返回,如果不是继续往下走

在这里会去判断当前正在进行如果不是正在使用,并且需要释放,则会把当前连接释放,之后继续回去初始化一个新的this.exchangeFinder = new ExchangeFinder(this, connectionPool, createAddress(request.url()),call, eventListener);

BridgeInterceptor

对request的header进行一波处理

CacheInterceptor
  • 根据request的url从cache里获取出response,根据这个request和response生成一个CacheStrategy,在这里面会对request和response进行一些添加header的处理,下面拿到的networkRequest和cacheResponse是处理之后的,
  • cache.trackResponse(strategy)用于记录创建此缓存之后总共发送的请求数量
  • 如果cacheResponse为空则关闭缓存
  • 如果被禁用了网络并且缓存是空则返回504报错
  • 如果networkRequest是空则直接返回一个根据cacheResponse生成的body为空的response
ConnectInterceptor
  • 判断是否可以新建连接,当前连接是否正打开着,如果是则抛出异常
  • 找到一个连接,可能是新建的,可能是已经存在的,如果是已经存在的还需进行是否已经关闭之类的校验
CallServerInterceptor

先把request写入,再取应答组成response

Response

CacheInterceptor
  • 如果有缓存并且应答的code是HTTP_NOT_MODIFIED,则将缓存与应答的header合并,并将cache放入cacheResponse,应答放入networkResponse,生成一个新的response,然后把InternalCache里的当前cacheResponse替换成新的
  • 如果有缓存并且code不是HTTP_NOT_MODIFIED则把缓存的body清空
  • 拿networkResponse和cacheResponse生成一个新的response
BridgeInterceptor
  • 把cookie保存下来
  • 如果用到了gzip则把数据进行处理,把gzip去掉重新生成response返回
RetryAndFollowUpInterceptor

如果之前的应答不为空,则把之前的应答加到priorResponse里,获取response的code判断是否需要重发,如果需要则继续循环
