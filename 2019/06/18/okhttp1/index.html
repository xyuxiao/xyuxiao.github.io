OkHttp源码阅读

发表于 2019-06-17 | 更新于 2019-07-03 | 分类于 Android | 阅读次数：

概述

在进行请求之前需要先new一个OkHttpClient,OkHttpClient内部是用的构建者模式初始化基础配置

需要先new一个Request,Request内部也是用的构造者模式实现的，而看了Request的源码就会发现Request没有public的构造函数,只有这么一个

在Request构建结束之后在发请求之前需要new一个Call,不管是异步(enqueue)还是同步(execute)都是需要的

在Call创建完成之后就需要把请求加入调度了

execute

先用synchronized加一个对象锁,看下executed是否为true,如果是true说明这个Call已经被执行过了,抛出异常,没有还没有被执行,则把executed的值置为true

同步的任务会用到AsyncTimeout,而AsyncTimeout是继承于TimeOut的一个异步超时类,主要是使用了一个WatchDog的守护线程在后台进行监听超时

把当前任务加到调度器的同步任务队列中

把自定义的拦截器和默认的拦截器都加进List传入RealInterceptorChain

  • 调用chain.proceed(request)
  • 调用第一个拦截器的interceptor.intercept(chain)
  • Request userRequest = chain.request();从chain()里获取到Request,然后userRequest.newBuilder(),获取到一个Request的Builder对象,在这里对Request进行操作,处理完成之后继续调用proceed并把Request.Builder.build之后获取到的处理过的Request传进去
  • 递归所有拦截器
  • 所有拦截器都调用完成后,chain.proceed返回的Response会在当前拦截器里调用response.newBuilder().request(userRequest)重新获取Response的Builder,并在当前拦截器进行处理,并在最后return responseBuilder.build(),return一个Response回去,继续做上一个拦截器的Response处理工作
  • 直到处理完所有拦截器的Response,然后把最后处理成功的Response返回回去

enqueue

前面和execute基本一样,也是先看Call是否执行过,对比较而言就是少了一个超时的守护线程

把AsyncCall添加到等待执行的异步请求队列readyAsyncCalls中,然后执行promoteAndExecute

遍历readyAsyncCalls,如果正在运行的异步请求队列小于最大值并且每个主机的最大请求值也小于最大值,则把等待的异步请求放入executableCalls和runningAsyncCalls,并把状态设为正在运行,再遍历executableCalls,调用executeOn方法(executorService是在当前调度器里新建的线程池)

AsyncCall是继承了NamedRunnable的而是实现了Runnable的

线程池执行Runnable对象会去执行run方法,而NamedRunnable的run方法中执行了execute,所以最后会调到AsyncCall的execute方法中

这里面操作就更同步差不多了,首先开启一个监听超时的守护线程,然后进行getResponseWithInterceptorChain操作,具体操作可以看execute的说明,在结束拿到Response之后执行传进来的Callback的回调
