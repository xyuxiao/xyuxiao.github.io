本地仓库

新建仓库

git init

在当前文件夹里调用git init会在当前文件夹初始化一个仓库, 会有一个.git的目录

在仓库目录里新建一个readme.txt文件

显示工作目录和暂存区的状态

git status

把文件添加到git仓库

git add readme.txt

这时候再查看状态就显示readme.txt是一个新文件了

提交到本地仓库

git commit -m “create readme”

-m 后面是跟的提交说明, 就是在之后查看提交记录(git log)的时候的说明

从图片中可以看到, 1 file changed, 0 insertions(+), 0 deletions(-)

  • 1 file changed 1个文件变动
  • 0 insertions(+) 0行添加
  • 0 deletions(-) 0行删除

这时候可以在文件中添加几行文字, 比如加个1234567, 再执行git add, gitcommit会看到1 file changed, 7 insertions(+)

  • 1 file changed 1个文件变动
  • 7 insertions(+) 7行添加

再在文件中删除几行文字, 再执行git add, git commit会看到1 file changed, 1 insertion(+), 5 deletions(-)

  • 1 file changed 1个文件变动
  • 1 insertion(+) 1行添加
  • 5 deletions(-) 5行删除

查看提交记录

git log
git log –graph –pretty=oneline –abbrev-commit

git add可以添加多个文件git add file1.txt file2.txt file3.txt 一起跟在后面就可以了

这时候继续在readme文件中添加4567890, 在修改文件之后查看git status

在这里我们只能看到这个文件变动了, 却不能看到他变动了什么, 这时候可能会想看他变动了什么

查看文件变动

git diff readme.txt

从图片可以看出减少了几行, 又新增了几行, 分别改动的是什么, 这时候要提交的话从之前git status可以看到no changes added to commit (use “git add” and/or “git commit -a”), 这时候如果想要提交就又需要先git add, git commit或者直接git commit -a

在add之后查看git status, 可以看到modified: readme.txt, 这时候就可以git commit了

这时候再执行git status 查看状态, 会看到working tree clean, 说明没有文件有变更了

这时候如果再执行git log查看提交记录就可以看到刚才提交的所有记录

从图中可以看到在commit之后面会有一串例如ba7c1a77267335c66288f9835507ff78dc04f25e这样的字符串,这串字符串是git的commit id, 这个id用于识别提交记录,而下面分别又有提交提交的人的信息和提交时间还有提交的时候-m后面带的信息

  • 当提交记录最后出现:或者END的时候可以用q退出
  • 提交人信息可以用git config –list

查看配置信息

git config –list

设置用户信息

git config –global user.name “xyuxiao”
git config –global user.email “xyuxiao_zyx@163.com“
git config user.name “xyuxiao”
git config user.email “xyuxiao_zyx@163.com“

–global是用于设置全局的属性的, 当你设置了–global之后所有新建的仓库的name和email都会是你设置的值,但是你也可以用下面没有–global的命令单独设置一个仓库的name和email

清除配置信息

git config –global –unset

版本回退

git reset –hard HEAD^
git reset –hard HEAD~1
git reset –hard ba7c1a

从刚才git log的图片中可以看到最上面也是我们最新的提交记录的commit id 后面跟了一个HEAD -> master 这说明当前master分支的当前版本在这个提交记录上(HEAD用于表明当前版本, 分支是什么后面会说), 这时候我们可以调用git reset来回退到之前的版本, 具体回退到哪个版本可以用HEAD^来表示,一个^标示一个版本, 当然如果版本太多的话也不可能写太多的^这时候就可以用HEAD~1来表示, ~后面跟的数字表示要回退几个版本

比如说我们这时候想回退到新增三个文件这次提交也就是diff测试之前,这时候就可以用git reset –hard HEAD^^

从HEAD is now at 123fbfb 新增三个文件 可以看出当前的HEAD已经回到了新增三个文件,也就是我们刚才想要回到的地方,从文件内容也可以看到我们回来了, 这时候查看git log就可以看到提交记录到这里就结束了

这时候我们来试试看另一种回退方式,比如我们想要回退到减少四行内容这个提交上,这时候我们刚才新增的file1, file2, file3应该就没了, 提交记录也应该回来

这时候如果我们想回到最新的版本,我们可以网上查看刚才最早的没有回退之前的git log,这时候我们可以看到最新的commit id是ba7c1a77267335c66288f9835507ff78dc04f25e,这时候我们就可以用这个commit id来进行git reset –hard ba7c1a

查看使用过的命令

git reflog

当找不到记录的时候也可以使用git reflog来查看使用过的命令,这时候也可以找到某次提交的commit id

没有提交的文件撤销

git checkout – readme.txt

当有的时候修改了一波, 突然不想要了, 这时候就可以用git checkout – readme.txt把文件回退到上次git add或者git commit的状态

  1. 当前如此图数字是到13的
  2. 修改为到7
  3. git checkout – readme.txt
  4. 查看数字又回到了13
  5. 再改到9
  6. git add readme.txt
  7. 再改到5
  8. git checkout – readme.txt
  9. 查看数字回到了9

从这里可以看出, git checkout – readme.txt是把状态回退到了把文件放入暂存区之前, 也就是把工作区的文件给回退了而不会去动暂存区里的文件, 而调用了git add之后文件就会被放到暂存区, git commit也只会把暂存区的文件上传

已经放入暂存区的文件撤销

git reset HEAD readme.txt

有的时候文件如果已经被git add放入暂存区了,这时候就需要用到git reset HEAD readme.txt来拯救了就比如说上一步中已经git add的那次修改, 就是把数字改到9那次

  1. git status 查看状态是已经add进缓存区的
  2. git reset HEAD readme.txt
  3. 查看状态已经回到了工具去, 可以查看diff, 已经有修改内容
  4. git checkout – readme.txt
  5. 查看数字回到了13

有的时候需要删除文件了, 这时候会直接把文件给删除了, 比如想删掉之前加了什么都没写的file3.txt, rm file3.txt

这时候用git status 查看的时候会提示file3被删除了,需要调用git rm file3.txt, 把它从仓库中删除, 当然如果调用了rm, 这次改动也就存入了暂存区了

远程仓库

与远程仓库连接

git remote add origin https://gitee.com/xyuxiao/git_test.git

本地仓库可以与远程仓库连接, 需要先创建一个远程仓库, 这里以gitee为例, 创建完成仓库之后会看到这么个界面

照着提示的命令输入git remote add origin https://gitee.com/xyuxiao/git_test.git即可把本地仓库与远程仓库连接

推送代码到远程仓库

git push origin master
git push -u origin master

在本地仓库与远程仓库连接成功之后就需要把本地的修改推到远程仓库, 这时候就可以用git push -u origin master把当前的仓库推送上去, 这是把master分支推送了上去, -u 是为了把本地的master分支和远程的master分支关联起来, 平时推送是不需要的

从远程仓库拉取代码

git clone https://gitee.com/xyuxiao/git_test.git

既然有推送上去, 那肯定也是有拉取代码的, 如果需要拉去代码那就可以用git clone https://gitee.com/xyuxiao/git_test.git

查看远程仓库信息

git remote
git remote -v

分支管理

新建分支

git checkout -b develop
git branch dev
git checkout dev

可以用git checkout -b develop 来新建一个分支, 这个分支的名字叫develop, -b是为了创建并切换到当前分支, 也可以分开写

查看远程分支

git branch -a

从远程仓库拉取分支

git checkout -b dev origin/develop

刚clone下来的新仓库, 只有一个master分支, 如果这时候需要在develop分支上进行修改, 就需要从远程仓库再把develop分支拉下来

查看当前有哪些分支

git branch

如果想查看当前有哪些分支, 可以用git branch来查看, 在列出的分支中, dev前面有个*, 这说明我们当前正在dev分支上

合并分支dev到当前分支

git merge dev

在一个分支上做操作的时候是不会影响到别的分支的, 比如

  1. 在dev分支上新增一个文件file.txt并把他提交了
  2. 在dev分支上查看提交记录会发现有这个提交
  3. 切换到master分支
  4. 查看提交记录会发现没有刚才新增file.txt文件的提交, 这是因为已经不是同一个分支了, dev分支修改的在master上是不会体现的
  5. 如果想让dev的提交在master上体现, 就需要把dev分支合并到master

仔细看上图提交日志的新增file和rm test两次提交, 在他们的commit id右边会有分支字样, 这是当前哪个分支正在这条提交记录

删除分支

git branch -d dev
git branch -D dev

在一个分支完成开发之后, 就会去把分支删除, 不然分支越积越多会不好管理, 并且长期不维护的分支留着也没意义

有的时候当开发一个分支到一半, 这个开发的功能突然没用了, 又因为什么关系不能保存下来, 需要删掉, 这时候如果直接去删除分支会提示没有合并不能删除, 如果一定要删需要用-D而不能用-d

删除远程分支

git push origin –delete develop

冲突解决

有的时候合并分支会出现在这个分支改了这个东西, 在另外的分支也改了这个东西, 那就会有冲突, 这时候就需要解决冲突

  1. 新建分支dev并来到dev分支
  2. 打开readme.txt并修改数字到7
  3. 提交当前的修改到dev分支
  4. 切换到master分支并且修改数字到18
  5. 提交当前的修改到master分支
  6. 这时候去调用git merge dev进行合并, git会提示我们有冲突, 需要解决
  7. 可以用git status查看当前冲突的问题, 如图所示可以看到readme.txt文件是存在冲突
  8. 打开readme.txt可以看到<<<<<<<, =======, >>>>>>>这样的符号, <<<<<<<和=======中间是当前分支的, =======和>>>>>>>中间是合并过来的分支的, 就是在这里面冲突了, 可以根据自己的需要进行修改
  9. 解决冲突之后重新提交一次代码, 可以在git log中看到在dev分支上的提交和在master分支上的提交现在在master分支上都可以看到提交记录了

远程冲突解决

有的时候修改了东西, 这时候需要push到远程仓库, 如果这时候别人也改了这个文件, 那push的时候就会报错了

根据git提示, 这时候需要git pull先拉一下

拉好之后查看git status会看到提示文件冲突了

解决完冲突commit,push就可以了

获取最新的更新

git fetch

有的时候可能想看看别人有没有上传东西, 但是还没确定的情况下不想直接拉下来, 这时候就可以使用git fetch获取最新的提交, 查看提交记录

存储

git stash
git stash save “改到14”
git stash apply
git stash drop
git stash clear

当有的时候改了一半需要做别的事情了, 这时候可以使用git stash把当前修改的存储起来, 等改好新的之后再把此次存储的内容来放出来

  1. 比如我现在在master分支修改了readme.txt文件
  2. 这时候我又要去develop改一波别的东西,就可以用git stash先把目前改的存起来, 当然为了便于寻找, 可以为此次存储赋值一个名称, 这时候就可以用git stash save “改到14”, 这时候再用git stash list 查看当前存储的stash的时候就可以看到我们刚存储的那次, 并且git status也表示没有修改了
  3. 当下次回来需要拿回之前的修改, 这时候就可以用git stash apply 来应用最近的一次保存
  4. 但是git stash apply 并不会把此次存储删除, 需要自己调用git stash drop来删除
  5. 这时候假如我们修改了一次readme.txt到了0, 存储起来名字叫修改到0, 另外一次修改到了20, 存储起来名字叫修改到20, 因为修改到20是后面修改的, 他会在前面出现, 这时候我们如果直接使用git stash apply只会应用最新的一次, 也就是修改到20, 如果我们现在想应用修改到0的这次修改, 我们就可以在git stash apply后面跟上存储的id, 当然删除也一个道理



  6. 如果我们想清空所有的stash, 还可以使用git stash clear

标签管理

打标签

git tag v1.0
git tag -a v1.2 -m “我是解决冲突的tag”


-a 表示tag名称, -m 表示本次提交的备注

查看标签

git tag


标签默认是打在当前提交上的, 如果想要打在之前的提交上, 可以在标签名称后面跟上commit id

查看标签信息

git show v1.0

删除标签

git tag -d v1.0

推送标签到远程

git push origin v1.1
git push origin –tags

推送tag到远程可以推送单独一个tag, 也可以一次性推送所有tag

查看查看标签

git ls-remote –tags origin

删除远程标签

git push origin –delete tag v1.1
git tag -d v1.1
git push origin :refs/tags/v1.1

git删除远程tag可以直接删除, 或者本地删除推送一个空的tag上去
